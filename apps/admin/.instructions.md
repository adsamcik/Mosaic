# Frontend (React 19 + Vite) - Scoped Instructions

> These instructions apply specifically to files in `apps/admin/`

## Technology Stack

- **Framework:** React 19 with Vite
- **Language:** TypeScript (strict mode)
- **State:** Zustand for global state, React Query for server state
- **Styling:** Tailwind CSS utility classes
- **Workers:** Comlink for Web Worker communication
- **Database:** SQLite-WASM with OPFS for local storage
- **Crypto:** libsodium-wrappers-sumo (via crypto worker)

## Strict Constraints

### React 19 Patterns
- Use Functional Components and Hooks exclusively
- NO Class Components
- Use `useTransition` for non-urgent updates
- Use `Suspense` boundaries for async operations

```tsx
// ✅ Good: Functional component with hooks
export function Gallery({ albumId }: GalleryProps) {
  const [isPending, startTransition] = useTransition();
  const photos = usePhotos(albumId);
  
  return <div>{/* ... */}</div>;
}

// ❌ Bad: Class component
class Gallery extends React.Component { }
```

### File Naming
- Components: `PascalCase.tsx` (e.g., `PhotoGrid.tsx`)
- Utilities: `kebab-case.ts` (e.g., `api-client.ts`)
- Hooks: `use-kebab-case.ts` (e.g., `use-photos.ts`)
- Tests: `component-name.test.tsx` or `utility.test.ts`

### TypeScript Rules
- NO `any` types—use `unknown` with type guards
- Explicit return types for exported functions
- Use discriminated unions for state machines
- Prefer `interface` over `type` for object shapes

```typescript
// ✅ Good: Discriminated union
interface LoadingState { status: 'loading' }
interface SuccessState { status: 'success'; data: Photo[] }
interface ErrorState { status: 'error'; error: Error }
type PhotosState = LoadingState | SuccessState | ErrorState;

// ❌ Bad: any type
function processData(data: any) { }
```

### State Management
- **Zustand:** Global app state (user session, settings)
- **React Query:** Server state (API calls, caching)
- **useState:** Local component state only

```typescript
// Zustand store
export const useSettingsStore = create<SettingsState>((set) => ({
  theme: 'system',
  setTheme: (theme) => set({ theme }),
}));
```

### Web Workers & Crypto
- ALL crypto operations run in the crypto worker
- Use Comlink for type-safe worker communication
- NEVER access `sodium` directly in React components

```typescript
// ✅ Good: Crypto through worker
const cryptoWorker = wrap<CryptoWorker>(
  new Worker(new URL('../workers/crypto.worker.ts', import.meta.url), { type: 'module' })
);
const decrypted = await cryptoWorker.decryptShard(envelope, key);

// ❌ Bad: Crypto in component
import sodium from 'libsodium-wrappers-sumo';
const decrypted = sodium.crypto_aead_xchacha20poly1305_ietf_decrypt(...);
```

### Memory Safety
- Zero sensitive buffers after use: `sodium.memzero(key)`
- Clear session keys on logout
- Never store keys in localStorage (use sessionStorage with encryption)

### Error Handling
```tsx
// ✅ Good: Specific error handling
try {
  await cryptoWorker.decryptShard(envelope, key);
} catch (error) {
  if (error instanceof DecryptionError) {
    logger.error('Decryption failed', { shardId });
    throw new UserFacingError('Could not decrypt photo');
  }
  throw error;
}

// ❌ Bad: Swallowing errors
try {
  await decrypt();
} catch (e) {
  console.log(e);
}
```

### Styling
- Use Tailwind CSS utility classes
- NO inline styles or CSS-in-JS
- Use CSS custom properties for theming

```tsx
// ✅ Good: Tailwind classes
<div className="flex items-center gap-4 p-4 bg-gray-100 dark:bg-gray-800">

// ❌ Bad: Inline styles
<div style={{ display: 'flex', alignItems: 'center' }}>
```

### Logging
- Use the centralized `logger` utility
- NEVER use `console.log` directly in production code
- NEVER log sensitive data (keys, passwords)

```typescript
import { logger } from '@/lib/logger';

// ✅ Good
logger.info('Photo uploaded', { photoId, albumId });
logger.error('Upload failed', { error: error.message });

// ❌ Bad
console.log('Uploading with key:', key);
```

## Testing Requirements

- Use Vitest with React Testing Library
- Mock workers and API calls
- Test user interactions, not implementation details

```typescript
import { render, screen, fireEvent } from '@testing-library/react';
import { describe, it, expect, vi } from 'vitest';

describe('UploadButton', () => {
  it('should trigger file picker on click', async () => {
    render(<UploadButton />);
    
    const button = screen.getByRole('button', { name: /upload/i });
    await fireEvent.click(button);
    
    expect(mockFileInput).toHaveBeenCalled();
  });
});
```

## Performance Guidelines

- Use `TanStack Virtual` for lists with 50+ items
- Lazy load images with `loading="lazy"`
- Use `React.memo` sparingly—only when profiling shows benefit
- Debounce search inputs (300ms)

## Accessibility

- All interactive elements must be keyboard accessible
- Use semantic HTML (`<button>`, `<nav>`, `<main>`)
- Provide ARIA labels for icon-only buttons
- Ensure 4.5:1 color contrast ratio
