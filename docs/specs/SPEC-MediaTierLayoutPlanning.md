# SPEC: Media Tier Layout Planning

## Status

Locked at v1. Implemented in `4dc18ac` (`feat(media): plan canonical tier
layouts`). `plan_tier_layout` returns canonical thumbnail (256×256, fit,
no-upscale), preview (1024×1024, fit, no-upscale), and original (validated
normalized dimensions) `TierDimensions` and is consumed by the tier generation
adapter and the Android UniFFI media bridge.

## Scope

This slice adds dependency-free, deterministic planning for Mosaic's canonical media tiers. It accepts already-normalized display dimensions and returns the target dimensions for thumbnail, preview, and original tiers.

The slice intentionally does not add image codecs, pixel decode/encode, file I/O, platform-specific behavior, backend/web/Android scaffolding, or new cryptography.

## Data Flow

Inputs:

```text
width: u32   // normalized display width from media inspection/orientation policy
height: u32  // normalized display height from media inspection/orientation policy
```

Output:

```text
TierLayout {
  thumbnail: TierDimensions { tier: ShardTier::Thumbnail, width, height },
  preview:   TierDimensions { tier: ShardTier::Preview,   width, height },
  original:  TierDimensions { tier: ShardTier::Original,  width, height },
}
```

The tier order is always thumbnail → preview → original. The original tier always uses the validated normalized display dimensions as-is.

## Sizing Policy

| Tier | Bounds | Policy |
|------|--------|--------|
| Thumbnail | 256×256 | Fit inside bounds, preserve aspect ratio, never upscale |
| Preview | 1024×1024 | Fit inside bounds, preserve aspect ratio, never upscale |
| Original | Validated normalized dimensions | Preserve as-is |

Integer math is used for all scaling. The dominant dimension is clamped to the tier bound, and the other dimension is computed with deterministic floor division, clamped to at least 1 pixel for valid extreme aspect ratios.

This approved media-core plan is intentionally limited to canonical layout planning for the Rust/Android upload path. It does not implement historical web adaptive-thumbnail sizing or embedded manifest thumbnail policy.

## Validation

`plan_tier_layout` reuses the same dimension validation as media inspection:

- width and height must be non-zero,
- `width × height` must not exceed `MAX_IMAGE_PIXELS`,
- violations return `MosaicMediaError::InvalidDimensions`.

## Zero-Knowledge Invariants

The planner never receives or emits plaintext media bytes, metadata, encryption keys, or server-visible content. It only processes dimensions and public tier identities. No server boundary, manifest encryption, key derivation, nonce generation, or shard encryption behavior changes in this slice.

## Verification Plan

- Landscape 4032×3024 plans 256×192 thumbnail, 1024×768 preview, 4032×3024 original.
- Portrait 3024×4032 plans 192×256 thumbnail, 768×1024 preview, 3024×4032 original.
- Square 3000×3000 plans 256×256 thumbnail and 1024×1024 preview.
- Small 128×96 source is not upscaled for thumbnail or preview.
- Non-divisible dimensions use deterministic integer floor scaling without floating-point assertions.
- Zero and over-budget dimensions return `InvalidDimensions`.
- Tier identities and tier order are thumbnail → preview → original.
- An orientation-swapped JPEG generated by existing test helpers is inspected first, then planned from normalized display dimensions.
